# Editor Ajar — Performance Budget & Methodology

> **Status:** Draft v0.1 · Companion to [SPEC §5](SPEC.md#5-non-functional-requirements-the-headline)
> and [TESTING](TESTING.md). Performance is a feature; this doc makes it measurable.

Performance and stability are *the* reason Editor Ajar exists. The targets in SPEC §5 are not
aspirations — they are **merge gates** (ADR-0011, ADR-0012). This document defines the reference
hardware, how each target is measured, and how regressions are caught.

## 1. Reference machine & conditions

- **Reference HW:** Apple Silicon M1 Pro, 16 GB RAM, internal SSD, macOS 14+.
- **Secondary tier (tracked, not gated):** M-series base chip, 8 GB; Intel is best-effort.
- **Conditions:** AC power, no thermal throttling at start, display at 60 Hz, release build.
- **Test media:** a committed-by-reference set (small clips + generated synthetic media) so
  benchmarks are reproducible. Large media is fetched/generated by `scripts/`, never committed.

## 2. Budgets (authoritative copy in SPEC §5)

| Metric | Target | How measured |
|--------|--------|--------------|
| Cold launch → interactive | ≤ 1.5 s | signpost from `main` to first interactive frame |
| Open 50-clip project | ≤ 2.0 s | signpost around document open |
| Playback 1080p30, 4 layers + 2 fx | real-time, **0 dropped frames** | frame-presented vs. expected count over 60 s |
| Playback 4K30, 2 layers + chroma key | real-time | as above |
| Scrub/seek latency | ≤ 1 frame / ≤ 50 ms | timestamp from seek call to correct frame presented |
| UI interaction | 60 fps, no hitch > 16 ms | CADisplayLink frame intervals during scripted drag |
| Keystroke → edit applied | ≤ 50 ms | signpost from event to model commit + redraw |
| Export 1080p30 H.264, 5 min | ≥ 3× real-time | wall-clock vs. timeline duration, HW encode |
| Idle memory (10-min 1080p) | ≤ 1.5 GB | footprint sample after settle |
| Idle CPU (paused) | < 3% | sampled over 10 s paused on timeline |
| Proxy gen (1080p) | ≥ 5× real-time | wall-clock vs. source duration |

**Throughput bench metrics** (`ajar bench`, report-only ×-real-time — unit `"x"`, unbudgeted since
absolute throughput is hardware-encoder-bound; compared to the SPEC §5 targets on the reference
machine, not gated on hosted runners):

| Bench metric slug | Requirement | Target | Notes |
|-------------------|-------------|--------|-------|
| `export-throughput-1080p30-h264` | NFR-PERF-008 | ≥ 3× | H.264 1080p30 export through the real `ExportSession`; capability-`skipped` (sentinel value 0) on encoder-less runners |
| `proxy-generation-throughput-1080p` | NFR-PERF-011 | ≥ 5× | ProRes-Proxy generation through `ProxyGenerationSession`; ProRes encodes on CI, never skips |

## 3. Performance budgeting per frame

At 30 fps the whole decode→composite→present chain has **~33 ms**; we target headroom so the
interactive layer never starves:

- Decode (HW, amortized via prefetch): off the critical present path.
- Render-graph build (CPU, `AjarCore`): target < 1 ms for typical sequences.
- GPU composite (per layer + effects): the bulk; measured per node type.
- Present + sync: minimal.

Each effect/transition shader carries a **per-node GPU cost budget** recorded in its benchmark;
adding an effect that blows the frame budget at the target spec fails CI.

## 4. Methodology

- **Instrumentation:** `os_signpost` intervals around launch, open, edit-commit, seek, frame
  present, and export. A `--bench` mode in the `ajar` CLI runs headless renders for repeatable
  GPU timing without UI noise.
- **Counters:** frames expected vs. presented (dropped-frame count must be 0 for gated specs),
  GPU frame time, peak/steady memory, CPU%.
- **Repeatability:** N warm-up iterations discarded, then K measured; report median + p95.
- **Regression gate:** the benchmark suite stores a baseline; a PR that regresses a gated metric
  beyond its noise band (default 5%) **fails** (ADR-0011). Baselines update via an explicit,
  reviewed commit.

## 5. Profiling playbook (for the agent and contributors)

1. Reproduce with the `ajar --bench <project> <metric>` harness first (no UI variance).
2. For GPU: Metal System Trace / GPU capture; check for CPU-GPU sync stalls and readbacks on the
   hot path (there should be none).
3. For CPU: Time Profiler; confirm the main thread is idle during playback and the audio thread
   is allocation-free.
4. For memory: Allocations + Leaks; confirm cache eviction holds budgets and the soak test is
   flat.
5. Record the finding and the fix's measured delta in the PR; update the baseline only if the
   change is an intentional, justified shift.

## 6. Anti-patterns that fail review

- CPU readback of GPU textures on the playback path.
- Allocation, locking, or Obj-C/Swift dynamic dispatch on the audio render thread.
- Synchronous disk or network I/O on the main thread or playback pipeline.
- Force-unwraps / `try!` (also a stability failure, NFR-STAB-003).
- "Works on my M3 Max" — targets are defined on the **reference machine**.
